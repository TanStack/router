# @tanstack/remix-router (experimental)

A binding from TanStack Router's framework-agnostic core to Remix 3's component runtime (`@remix-run/ui`). Lets you use type-safe TanStack routing — loaders, search params, route trees — inside a Remix 3 app.

> [!WARNING]
> This package is a working spike against the **Remix 3 beta**. Both Remix 3 and this binding will change. Do not depend on it from production code yet.

## Why

Remix 3 dropped React for its own JSX runtime and ships as a "Demand Composition" toolkit — every package is independently usable. TanStack Router's core is already framework-agnostic (the React/Solid/Vue bindings each plug a reactivity layer into a common `RouterCore`). That makes Remix 3 the natural fourth peer: drop the TSR core into `@remix-run/ui` for the view, keep `@remix-run/fetch-router` for the server (or swap it for the included `createRouterHandler`), and compose the rest of Remix 3's modular packages (`session`, `auth`, `form-data-parser`, `file-storage`, …) around it.

## Quick start

```tsx
// app/router.ts
import { createRouter } from '@tanstack/remix-router'
import { routeTree } from './routeTree.gen'

export function getRouter() {
  return createRouter({ routeTree })
}
```

```tsx
// app/client.tsx
import { run } from '@remix-run/ui'
import { hydrateStart, StartClient } from '@tanstack/remix-start/client'
import { createRoot } from '@remix-run/ui'
import { getRouter } from './router'

const router = await hydrateStart(getRouter)
run({ loadModule: (url, exportName) => import(url).then((m) => m[exportName]) })
createRoot(document.body).render(<StartClient router={router} />)
```

```tsx
// app/server.ts (with @tanstack/remix-start + Vite plugin)
// The default-entry server is auto-generated by the plugin and uses
// `defaultStreamHandler` from `@tanstack/remix-start/server`. You
// don't normally need to write this yourself.
```

## Design notes

The `Handle` model in `@remix-run/ui` is a setup-once-render-many factory. There's no fiber stack, so React-style hooks don't translate. Instead, accessors take a `handle` and return getters that subscribe at setup-time and clean up via `handle.signal`:

```tsx
import type { Handle } from '@remix-run/ui'
import { useLoaderData } from '@tanstack/remix-router'

export function UserDetail(handle: Handle) {
  const readUser = useLoaderData(handle, { from: '/users/$id' })
  return () => {
    const user = readUser()
    return <div>Hello {user?.name}</div>
  }
}
```

`useLoaderData` registers a subscription on the route-match store, schedules `handle.update()` on change, and unregisters when `handle.signal` aborts.

### Reactivity model

`@remix-run/ui` re-runs the render function on `handle.update()` — coarser than React's per-fiber updates but simpler. To make sure subscriptions follow the right store across navigations (where a single component instance can be reused for different matches), the binding uses `subscribeDynamicStore`, which re-binds when the prop-derived store identity changes.

`<MatchContext>` and `<RouterContextProvider>` set their context inside the **render** function (not setup) so prop changes propagate through reused vnode instances.

### Function-as-children

Remix 3's JSX runtime can't pass function values through `children` — the `flatMapChildrenToVNodes` step rejects them. Components like `<Await>` and `<Block>` that traditionally take function children expose a `render` prop instead:

```tsx
<Await
  promise={data.slow}
  fallback={<p>loading…</p>}
  render={(slow) => <p>{slow.payload}</p>}
/>
```

The `children` prop still works for static vnodes.

### Document shell

`<RouterProvider>` renders only the body content. The document shell (`<html>/<head>/<body>` plus `<HeadContent>` and `<Scripts>`) is owned by `<StartServer>` from `@tanstack/remix-start/server`. On the client, hydration mounts into `document.body`, matching the body-content tree without an `<html>`-as-render-root mismatch.
